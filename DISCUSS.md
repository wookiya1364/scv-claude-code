## Round 1 — 첫 인상 (1 주일 사용 후)

**최우석 (중재자)**: 1 주일 써 보고 첫 느낌 한마디씩 부탁드립니다. 좋았던 점·아쉬웠던 점 한 문장씩 먼저요.

**홍지영 (옹호)**: 첫 인상은 회의적이었는데 막상 PLAN.md scaffold 가 강제하는 4 섹션 (Summary / Goals / Steps / Related) 이 리뷰 시간을 절반으로 줄였어요. 코드만 보고 "이게 왜 필요?" 묻던 시간이 사라짐.

**박민수 (반대)**: 도구 피로감 큽니다. Linear 에서 이미 ticket 쪼개고, GitHub PR 에서 이미 description 쓰는데 SCV 의 PLAN.md 는 그 둘과 **같은 정보의 세 번째 사본**이에요. DRY 위반.

**이수진 (중립)**: 4 명이 1 주 써서 PR 12 개 / promote 폴더 8 개 / archive 6 개 만들었어요. 머지 후 회귀 발견 사례는 0. 그런데 promote 작성에 평균 18 분 추가됨. 효익은 아직 모름.

**강도현 (극렬 반대)**: 근본 의심을 깔고 시작합니다. **Claude Code 라는 비결정적 도구로 코드 짜고, 그 코드를 Claude 가 만든 plugin 의 protocol 로 검증한다는 게 자기참조 무한루프**예요. AI 가 자기 자신의 quality gate 를 정의하는 건 reviewer 와 reviewee 가 같은 인격인 셈.

**정유나 (극렬 옹호)**: 강도현님 비판은 form 만 맞고 내용은 틀려요. SCV 의 quality gate 는 **TESTS.md 의 실행 방법** 이고 그건 결정적 bash 커맨드. AI 가 정의하는 게 아닙니다. PR 50 개/일 폭증 사례를 막을 유일한 메커니즘이 이거예요. 6 개월 안에 우리도 그 사례 됩니다.

**최우석 (중재자)**: 좋습니다. 첫 인상에서 의견 분기 명확하네요 — 형식 가치 vs DRY 위반 vs 데이터 부재 vs 자기참조 vs 미래 보장. 다음 라운드는 구체 마찰 지점으로.

## Round 2 — 실제 마찰 지점

**박민수 (반대)**: raw 에 회의록 던지는 문화. 우리 팀 4 명 중 1 명만 따라요. 나머지는 슬랙에 떠들고 끝남. SCV 가 강제하지 못하면 절반의 팀에서만 작동하는 도구.

**홍지영 (옹호)**: 그건 도구가 아니라 **onboarding 문제** 예요. 첫 주에 1 번 같이 PLAN.md 채워보면 다음부터 자연스러워져요. 우리 팀에선 2 일 후 모든 팀원이 raw 에 던지기 시작했음.

**강도현 (극렬 반대)**: TESTS.md 의 `## 실행 방법` 이 **모호하면 `/scv:work` 가 사용자에게 다시 묻고 멈춘다** — 이게 문서에 명시돼 있죠. 빠른 prototype 단계에선 치명적입니다. 우리는 모호한 테스트로 일단 머지하고 다음 sprint 에 정리하는 패턴을 자주 씁니다. SCV 는 그걸 막아요.

**정유나 (극렬 옹호)**: 그게 **정확히 SCV 의 의도** 입니다. 모호한 테스트 = 모호한 feature = 안 머지. 강도현님이 말한 "다음 sprint 에 정리하는 패턴" 이 통계적으로 절대 정리 안 되는 게 업계 흔한 패턴이고, 그게 누적돼서 PR 50 개/일 사례가 되는 겁니다.

**박민수 (반대)**: 통계 출처는요?

**정유나 (극렬 옹호)**: 그 회사 사례가 직접 데이터입니다. "다음에 정리한다" 는 약속이 누적된 결과가 6/10 명이 미검증 PR 올리는 상태.

**이수진 (중립)**: 작은 변경은 어떻게 하나요? 오타 수정·1 줄 hotfix 까지 promote 폴더 만들고 PLAN/TESTS 채우는 건 분명 과한 의례인데 SCV 문서에는 그 fast-path 가 명문화 안 돼 있어요.

**홍지영 (옹호)**: 그건 인정할 수밖에. 지금 PROMOTE.md 가 "작은 변경엔 promote 안 거쳐도 됨" 이라고 명시 안 합니다. 사람들이 1 줄 수정에도 promote 만들기 시작하면 형식 비용 폭증.

**강도현 (극렬 반대)**: 거기에 더해서 — **AI 와 대화로 PLAN.md 를 채우는 시간이 코딩 시간보다 길게 느껴져요**. 18 분 추가됐다는 데이터를 이수진님이 주셨는데 이건 평균이고 첫 회는 40 분 걸렸어요. 6 회 째에야 10 분 미만. 학습 곡선 가팔라요.

**홍지영 (옹호)**: 학습 곡선은 일회성 비용이고 18 분이 평균이면 PR 1 개당 **18 분** 들어가서 **리뷰 시간 30 분 절약** 한다면 12 분 순이익. 다만 이수진님 데이터로는 그 30 분 절약이 입증 안 됐죠.

**이수진 (중립)**: 맞아요. 1 주는 그 데이터 모으기 너무 짧음. 4 주는 봐야 해요.

**최우석 (중재자)**: Round 2 정리하면 — 마찰은 (a) 팀 문화 강제력 부재, (b) 모호한 테스트 차단의 양면성, (c) 작은 변경 fast-path 부재, (d) PLAN 작성 시간. 이 중 (c) 는 모두 동의하는 약점입니다. 다음 라운드는 SCV 의 가치가 무엇인지 정면으로.

## Round 3 — 가치 추출 (1 년 후 시점에서 보면?)

**정유나 (극렬 옹호)**: 1 년 후 archive 에 쌓인 200 개 PLAN+TESTS 는 **회사 자산** 입니다. 새 사람 입사 시 "결제 v2 가 어떻게 만들어졌나?" → archive 의 epic-pay-overhaul/ 폴더 7 개 PLAN.md 읽으면 1 시간에 맥락 파악. 지금 우리는 1 주일 onboarding 합니다.

**박민수 (반대)**: 200 개 PLAN.md 가 1 년 후에도 **현재** 인 보장이 없어요. 코드는 바뀌는데 PLAN 은 archive 라 영구 불변. 1 년 된 PLAN 을 보고 입사자가 헷갈릴 가능성이 더 큼.

**홍지영 (옹호)**: 그래서 supersedes 가 있는 거예요. 새 PLAN 에 `supersedes: [<old-slug>]` 박으면 옛 PLAN 은 archive 에 그대로 있되 `obsolete` 마킹. 입사자가 첫 줄 보고 "이건 폐기됨" 알 수 있어요.

**강도현 (극렬 반대)**: 그 supersedes 매번 선언이 부담입니다. 작성자가 "내가 뭘 대체하는지" 매번 정확히 알아야 함. 모르면 그냥 안 쓰고, 안 쓰면 아카이브가 거짓말로 가득.

**정유나 (극렬 옹호)**: 모르면 `/scv:regression` 이 회귀 실패 시점에 **3-way triage** 로 잡아요. supersedes 빠뜨렸어도 런타임에 사용자가 "obsolete" 선택하면 마킹됨. 이중 안전망입니다.

**이수진 (중립)**: 1 년 후 archive 가 누적되면 회귀 시간 폭증 우려는요? 200 개 archive 에 각 TESTS 가 30 초씩이면 100 분. 아무도 안 돌립니다.

**홍지영 (옹호)**: `--tag core` 같은 필터로 critical path 만 돌릴 수 있어요. 게다가 supersedes/obsolete 가 자동 skip 하니까 활성 회귀는 100 개 미만 유지될 가능성 높음.

**강도현 (극렬 반대)**: "유지될 가능성" 은 가설이지 데이터 아닙니다.

**정유나 (극렬 옹호)**: 그래서 dogfooding 하는 거고, v0.2.0 이 stable 하지 않다고 0.x 로 표시한 거예요. 1 년 데이터 모은 후 1.0 안정 API 확정.

**박민수 (반대)**: 1 년 동안 우리가 dogfood 라는 거네요. 회사 비즈니스 로직이 0.x 도구의 학습 데이터로 쓰임.

**최우석 (중재자)**: 박민수님 의견은 SCV 의 0.x SemVer 시그널과 **adoption 모드** 의 의미를 다시 강조해 주네요. SCV 가 "강제 안 함" 을 명시한 이유가 그거였죠. 다음 라운드는 결정·합의로.

## Round 4 — 합의·분기·결정 권장

**최우석 (중재자)**: 4 라운드 토론 종합하면 **합의 영역** · **의견 분기 영역** · **데이터 부족 영역** 세 가지가 명확합니다.

### 합의 영역 (전원 동의)

1. **PLAN/TESTS 형식이 리뷰 컨텍스트를 풍부하게 한다** — 박민수님조차 DRY 측면 비판이지 형식 자체를 부정하진 않음
2. **adoption 모드의 비강제성은 적절** — `status: N/A` 로 INTAKE 강제 안 하는 게 도입 마찰을 줄임
3. **작은 변경 (오타·1 줄 hotfix) 의 fast-path 가 문서에 명문화돼야 한다** — 이건 명백한 약점
4. **0.x SemVer 시그널** — 도구가 학습 단계임을 명시한 건 정직

### 의견 분기 영역 (해소되지 않음)

| 쟁점 | 옹호 측 | 반대 측 |
|---|---|---|
| 모호한 테스트의 차단 효과 | 의도된 quality gate | 빠른 prototype 단계에 치명 |
| Claude 와의 대화 시간 비용 | 학습 곡선 후 줄어듦 + 리뷰 시간 절약 | 입증된 데이터 부재 |
| supersedes 매번 선언 부담 | runtime triage 가 안전망 | 잊으면 archive 가 거짓말로 가득 |
| AI-of-AI 자기참조 | TESTS 는 결정적 bash, AI 가 quality gate 를 정하는 게 아님 | 메타 계층의 자기검증 본질적 한계 |

### 데이터 부족 영역 (4 주 이상 필요)

- 1 년 후 archive 회귀 시간이 실제로 100 분 폭증할지 vs supersedes/obsolete 로 100 개 미만 유지될지
- onboarding 시간 단축 (1 주 → 1 시간) 이 실제 입사자에게 작동하는지
- 팀 문화 강제력 부재가 도입을 좌초시키는지 vs 천천히 자율적으로 자리잡는지
- 18 분 PLAN 작성 시간이 30 분 리뷰 시간 절약을 정말로 만드는지

### 권장 결정 (중재자 종합)

| # | 권장 변화 | 근거 | 우선순위 |
|---|---|---|---|
| 1 | **PROMOTE.md 에 fast-path 명문화** — "오타 수정·1 줄 hotfix·outage hotfix 같은 변경은 promote 폴더 없이 직접 PR 권장. 적용 기준 명시" | 합의 영역, 명백 약점 | 🔴 높음 |
| 2 | **Onboarding 가이드 추가** — `docs/ONBOARDING.md` 또는 README 의 첫 주 walkthrough. 첫 promote 같이 만들어보는 시나리오 | 마찰 지점 (a), 도입 성공 결정 요인 | 🟡 중간 |
| 3 | **회귀 자산 관리 정책** — 1 년 후 회귀 시간 측정해 evict / 빈도 down-sample 기준 추가. 지금은 0 개 archive 라 결정 보류 | 데이터 부족 영역 | 🟢 낮음 (6 개월 후 재검토) |
| 4 | **SCV 사용 데이터 telemetry** (opt-in) — `/scv:work` 평균 시간, archive 누적량, 회귀 실패율 등을 익명 집계. 1.0 안정 API 결정 근거 | 데이터 부족 영역 모두 | 🟡 중간 (privacy 검토 필요) |
| 5 | **모호한 테스트 차단의 강도 조절** — `/scv:work` 에 "지금은 prototype 단계, 모호 OK" 옵션 (사용자 명시 선택 시) | 의견 분기 (강도현 vs 정유나) | 🟡 중간 (1 명 강한 반대 → 2 명 강한 옹호) |

### 비채택 권장 (의도적)

- **AI-of-AI 자기참조 회피** — 강도현님 형이상학적 비판은 흥미롭지만 SCV 의 실제 quality gate (TESTS 의 bash 실행) 는 결정적이라 본 비판이 적용 안 됨. 도구 변화 불요.
- **PLAN.md ↔ Linear ↔ GH PR 통합** — 박민수님 DRY 비판은 일리 있으나 통합은 큰 작업이고 0.x 단계에선 우선순위 낮음. 1.0 에 도전.
- **모호 테스트 강제 우회 (work.md 룰 변경)** — Pure 옹호 입장 기각. 모호 테스트가 절대 정리 안 되는 통계적 패턴을 깨는 게 핵심 가치이므로 강도 조절은 했지만 룰 자체는 유지. 정유나님 의견 채택.

## 마무리 발언

**박민수**: 1 줄 hotfix fast-path 가 명문화되면 SCV 시도 의지 50% → 70% 로 올라갑니다.

**홍지영**: telemetry opt-in 좋네요. 1.0 가는 길이 더 단단해집니다.

**이수진**: 4 주 후 리뷰 다시 잡겠습니다. 그때 데이터 들고 옵니다.

**강도현**: 자기참조 비판은 채택 안 됐지만 인정합니다. 단, "prototype 단계 옵션" 추가는 환영. 강제성 완화 방향이라.

**정유나**: 강제성 완화는 양보 안 받고 싶은데 1 명 강한 반대 + 1 명 일반 반대 있으니 받아들이겠습니다. 단, 옵션이 default 되면 안 됩니다.

**최우석 (중재자)**: 토론 종료. 권장 결정 5 개를 사용자에게 전달합니다. **🔴 #1 (fast-path 명문화) 가 이번 라운드의 가장 명확한 다음 액션** 이고, 나머지는 우선순위에 맞춰 단계적으로 결정하시면 됩니다. 의견 분기 영역의 데이터 (4 주 후 회고, 6 개월 후 회귀 시간) 가 모이면 다음 토론에서 명확해질 겁니다.
